# Backend — API Contract

This folder will eventually hold the FastAPI backend. Today it contains only
the API contract:

- `openapi.yaml` — full OpenAPI 3.0 specification. **Hand this file to the
  client team to generate Retrofit interfaces.**
- `app/api/schemas.py` — Pydantic source-of-truth models that the spec
  mirrors. Once the backend is scaffolded, FastAPI will auto-generate an
  identical OpenAPI doc from these.
- `requirements.txt` — placeholder; only Pydantic is needed to import the
  schemas. Full backend deps come later.

## Endpoints at a glance

All endpoints are under `/api/v1`.

| Method | Path | Surface | Purpose |
|--------|------|---------|---------|
| GET    | `/dlcs` | DLC | List available DLCs (currently one) |
| POST   | `/playthroughs` | Storyline | Start or resume a playthrough |
| GET    | `/playthroughs/{id}/scene` | Storyline | Fetch the current scene + state |
| POST   | `/playthroughs/{id}/decisions` | Storyline | Submit a decision (CHOICE or FREE_TEXT) |
| GET    | `/playthroughs/{id}/ending` | Storyline | Fetch the final ending after S5 |
| POST   | `/hermione/query` | Hermione | Ask Hermione's Notes (RAG hint) |

## Storyline flow (the typical session)

```
Client                                Backend
  │                                     │
  ├── POST /playthroughs ──────────────►│  body: { player_uuid, dlc_id }
  │◄──────────── 200 { playthrough, scene }
  │      (player can now read scene 1)
  │
  ├── POST /playthroughs/{id}/decisions ►│  body: CHOICE or FREE_TEXT
  │◄──── 200 { is_correct, delta_score, narrative_feedback, next_scene }
  │      (UI shows feedback, then the next scene)
  │
  │  ... repeat for scenes 2..5 ...
  │
  ├── GET /playthroughs/{id}/ending ───►│
  │◄──────────── 200 { ending_id, title, narrative, summary }
```

## Decision payload — discriminated by `mode`

The body of `POST /decisions` is one of two shapes, picked by the `mode` field:

```jsonc
// CHOICE
{
  "scene_id": "S1_THE_VISION",
  "mode": "CHOICE",
  "choice_id": "B"
}

// FREE_TEXT
{
  "scene_id": "S4_THE_BRINK_OF_DEATH",
  "mode": "FREE_TEXT",
  "text": "Cast Protego on Sirius and pull him back from the Veil."
}
```

OpenAPI represents this as `oneOf` with `discriminator: mode`. Most code
generators (including `openapi-generator-cli` for Retrofit) handle this
correctly out of the box.

## Hermione's Notes — separate surface

`POST /hermione/query` is independent of the storyline. It does not change
score, scene, or playthrough status. The player can call it at any time.

```jsonc
// request
{ "player_uuid": "...", "query": "What does the Veil do?" }

// response
{
  "answer": "An ancient archway whose veil sways without wind. Those who pass through never return.",
  "sources": [{ "lore_id": "veil_001", "title": "The Veil in the Death Chamber" }]
}
```

The backend prompts the LLM not to reveal correct decisions directly
(FR 1.6), so answers are hint-shaped rather than instructions.

## Identity & errors

- **No auth.** Every request that needs a player carries `player_uuid` —
  a UUID generated by the Android client on first launch and persisted in
  `SharedPreferences`.
- **Errors** use `ErrorResponse`:
  ```jsonc
  { "error_code": "PLAYTHROUGH_NOT_FOUND", "message": "...", "details": null }
  ```
  Standard HTTP status codes: `400` malformed, `404` missing resource,
  `409` state conflict (wrong scene, ending already reached, etc.).

## Generating a Retrofit client

```bash
# from the project root
npx @openapitools/openapi-generator-cli generate \
  -i backend/openapi.yaml \
  -g kotlin \
  -o ../android/app/src/main/java/generated \
  --additional-properties=library=jvm-retrofit2,serializationLibrary=kotlinx_serialization
```

(Adjust the output path to whatever the Android module expects. The Retrofit
2 + kotlinx-serialization combo is the most common modern pairing.)
